# 拆分 Propose 為多個 tasks 規格檔

從 `docs/Tasks/v*/` 下的 `propose-vX.X.*.md` 抓取子版本規劃，依序拆分產出多個 `tasks-vX.X.N.md` 任務規格檔，每檔對應一個子版本。

**必須遵守 `CLAUDE.md` 與 `docs/Design-Base/*` 的所有規範**（繁體中文輸出、任務文件回填格式、不過度抽象）。

## 使用方式

```text
/split-tasks-slot                         # 互動模式：列出所有 propose 供選擇
/split-tasks-slot <propose 檔案路徑>      # 直接指定來源
/split-tasks-slot <路徑> --dry-run        # 只顯示拆分計畫，不寫檔
```

範例：
- `/split-tasks-slot docs/Tasks/v1.2/propose-v1.2.0.md`
- `/split-tasks-slot docs/Tasks/v1.3/propose-v1.3.0.md --dry-run`

## 執行前置

1. 讀取根目錄 `CLAUDE.md`：遵守「任務文件回填」規則（commit 是事件紀錄、task doc 是階段真相來源）。
2. 讀取 `docs/Design-Base/00-overview.md`、`20-backend.md`、`21-database.md`：拆分時凡涉及 DB Migration、API、前端結構皆須引用這些規範。
3. **參考既有任務檔慣例**：產出格式須對齊 `docs/Tasks/v1.1/tasks-v1.1.*.md` 與 `docs/Tasks/v1.2/tasks-v1.2.*.md`，至少讀過其中一份作為樣板。
4. 若有提供參數，將其視為來源檔案路徑；否則進入互動模式（見「執行步驟 §1 選定來源檔案」）。
5. 所有輸出一律使用**繁體中文**。

## 執行步驟

### 1. 選定來源檔案

- **有參數**：驗證路徑存在且為 `.md` 檔。不存在則報錯結束。
- **無參數**：
  - 列出 `docs/Tasks/v*/` 下所有 `propose-*.md`
  - 透過 `AskUserQuestion` 工具供使用者選擇（每個 propose 為一個 option，最多 10 個）
  - 若候選超過 10 個，先依檔名版本號倒序取最新 10 個

### 2. 解析來源檔案

從來源檔案抽取：

| 項目 | 解析來源 |
| --- | --- |
| 大版本號（`vX.X`） | 檔名 `propose-vX.X.*.md` 或檔案首行標題 |
| 最終目標 | `> 最終目標：...` 引用行 或 §1「版本目標」首段 |
| 子版本清單 | **§3 子版本 Roadmap** 表格（必填）— 每列對應一個 `tasks-vX.X.N.md` |
| 各子版本範圍 | 表格欄位「主題」「性質」「時程估」「前後相依」+ 對應 §2 的子節引用 |
| 跨版本衝突 | §5 衝突檢查表（若有） |
| 驗收清單 | §6 驗收（若有）— 依分類或關鍵字對應到各子版本 |

**若來源檔案缺少 §3 Roadmap 表格，立即停止並告知使用者必須先補上，不得猜測拆分。**

### 3. 產出拆分計畫（Dry Run 預覽）

不論是否 `--dry-run`，**先產出並顯示**以下拆分計畫讓使用者確認：

```text
來源檔案：docs/Tasks/v1.2/propose-v1.2.0.md
大版本：v1.2
最終目標：<從 propose 抽取>

擬產出：
  1. tasks-v1.2.1.md  主題：... / 性質：... / 時程：...
  2. tasks-v1.2.2.md  主題：... / 性質：... / 時程：...
  3. tasks-v1.2.3.md  主題：... / 性質：... / 時程：...
  4. tasks-v1.2.4.md  主題：... / 性質：... / 時程：...

輸出目錄：docs/Tasks/v1.2/
既有檔案衝突：<列出任何已存在的 tasks-vX.X.N.md>
```

- 若 `--dry-run`：僅顯示上述資訊後結束，**不寫檔**。
- 若非 `--dry-run`：透過 `AskUserQuestion` 詢問是否執行（選項：`執行` / `取消`）；無回應則預設執行。

### 4. 處理檔案衝突

- 若目標 `tasks-vX.X.N.md` **已存在**：
  - **預設不覆寫**（skip）
  - 透過 `AskUserQuestion` 詢問處置方式（選項：`skip` / `overwrite` / `backup(.bak)`），無回應則 skip
- 已存在且狀態標題為「已完成」或「進行中」的檔案：**強制 skip**，僅在報告中提示使用者

### 5. 依模板產出每個 tasks-vX.X.N.md

每個檔案遵循以下結構（對齊 `docs/Tasks/v1.1/tasks-v1.1.*.md` 與 `docs/Tasks/v1.2/tasks-v1.2.*.md` 既有慣例）：

```markdown
# vX.X.N 任務規格：<主題>

> 前置：[<來源 propose 檔名>](<相對路徑>) §<對應章節>、[<相依任務檔>](<相對路徑>)（簡述為何相依）
> 後續依賴：<列出依賴本版的子版本，若無可省略整行>

## 版本目標

<從 propose §3 Roadmap 的「主題」欄位 + 對應 §2 子節首段，用 1-3 句話描述本版要交付什麼>

### 範圍內

<從 propose §2 的對應子節抽取要點，條列>

### 範圍外

<從 propose §1「範圍外」或 §2 標示「留給 vX.X+N / 未來版本」的項目對應抽取，並用 `（→ vX.X.N）` 指向後續版本>

---

## 前置現況

<從 propose 開頭「前置假設」或 §0 抽取本版開工前已具備的能力 / 既有檔案 / 已交付任務>

- <條列既有可用的元件 / 表 / 服務>
- <條列本版需引入但尚未存在的依賴（例：Redis、新套件）>

---

## 已確認決策

| # | 決策 | 結論 |
| --- | --- | --- |
| 1 | <從 propose §2 對應子節抽取的決策點> | <該決策的最終結論> |
| 2 | ... | ... |

---

## Phase 1：<分類，例 Backend / Migration / Frontend>

### 1-1 <子節，例 Migration / Schema / Service>

- [ ] <最小可實作的具體任務，能對應一個或數個檔案改動>
- [ ] ...

### 1-2 <下一子節>

- [ ] ...

---

## Phase 2：<分類>

### 2-1 ...

- [ ] ...

---

## Phase N：驗收

<從 propose §6 驗收抽取與此子版本相關的項目；保留 `[ ]` 讓實作後回填；若 propose §6 已分組（例「基本功能 / API 行為 / Sidebar」），保留分組>

- [ ] ...
- [ ] ...
```

#### 模板填寫紀律

- **不放「狀態」行**：未開始時不放，僅在實作期間 / 完成後才補（依 `CLAUDE.md §任務文件回填`）
- **Phase 編號用阿拉伯數字 + 子節 X-N**：對齊 v1.1.1 風格（`Phase 1：後端` / `### 1-1 Migration`）
- **checkbox 必須保留 `[ ]`**：拆分時所有任務項皆未完成
- **相依與衝突資訊融入**「前置」引用塊與 `## 前置現況`，**不**另開 `## 相依與衝突` 段
- **保留原文語氣**：抽取內容時沿用 propose 原句，不重寫總結變樣

#### 回填規則（產出時不寫進檔案，僅供使用者後續維護用）

- 任一範圍項完成 → 對應 checkbox `[ ]` 改 `[x]`
- 規格被推翻 → 保留 checkbox，後方補 `—（已改為 xxx，見 commit yyy）`
- 全部完成 → 頂部加狀態行 `> **狀態：已完成（commit xxx, YYYY-MM-DD）**`
- 部分完成 → 狀態行 `> **狀態：進行中（已完成 Phase N, 剩 ...）**`

### 6. 回報結果

寫檔完成後輸出摘要：

```text
✅ 已產出 N 個 tasks 檔：
  - docs/Tasks/v1.2/tasks-v1.2.1.md
  - docs/Tasks/v1.2/tasks-v1.2.2.md
  - ...
⏭  跳過 M 個既有檔（skip / backup）：
  - ...

下一步建議：
  1. Review 每個 tasks 檔的「範圍內」與「驗收」是否完整
  2. 從最底層子版本（無相依）開始開工
  3. 實作時依 CLAUDE.md §任務文件回填 規則即時更新狀態
```

## 注意事項

- **檔名嚴格為 `tasks-vX.X.N.md`**，全小寫、`-` 為分隔、`v` 為前綴、三段版號
- **不覆寫既有檔案**（尤其狀態為「已完成」者），有衝突一律走互動詢問
- **不得猜測子版本拆分**：若 propose 缺 §3 Roadmap，停止並告知補完才能拆
- **保留原文語氣**：抽取內容時盡量沿用 propose 原句，不自行重寫或總結變樣
- **checkbox 必須保留 `[ ]`**：拆分時所有驗收 / Phase 項皆為未完成狀態，不得預先打勾
- **不執行任何 Git 操作**：本指令只寫檔，不負責 commit / push
- **若 propose 為 extended 類型**（如 `propose-v1.1-extended.md`）：子版本號不假設從 1 起算，依 propose §3 Roadmap 表格列出的版本號為準（例 v1.1 extended 是從 v1.1.5 起）
- **資料來源**（內容抽取）：propose 為唯一來源；不得從其他檔案推斷子版本拆分
- **規範優先順序**（產出格式）：`docs/Design-Base/*` > `CLAUDE.md` > 其他（與 `CLAUDE.md §規範優先順序` 一致）
- **產出後不自動開啟檔案**、**不自動跑測試 / build**，僅產檔與回報
