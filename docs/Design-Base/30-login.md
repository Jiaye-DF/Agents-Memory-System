# 登入系統規範

## 認證機制

採用 **Access Token + Refresh Token** 雙 Token 架構：

| Token         | 存放位置           | 有效期   | 用途                     |
| ------------- | ------------------ | -------- | ------------------------ |
| Access Token  | HTTP Header Bearer | 15 分鐘  | API 請求認證             |
| Refresh Token | HttpOnly Cookie    | 7 天     | 換發新 Access Token      |

### 流程

1. 登入成功 → 後端簽發 Access Token（回應 body）+ Refresh Token（HttpOnly Cookie）
2. 前端每次 API 請求攜帶 `Authorization: Bearer <access_token>`
3. Access Token 過期 → 前端自動呼叫 `/api/v1/auth/refresh` 用 Refresh Token 換發新 Token
4. Refresh Token 過期 → 導回首頁重新登入
5. 登出 → 後端清除 Refresh Token Cookie，並將該 Token 加入 Redis 黑名單（剩餘有效期內拒絕使用）

### Token 規範

- 使用 **JWT**（`HS256` 或 `RS256`），payload 包含 `user_uid`、`exp`、`iat`
- **禁止**在 Token payload 中放入密碼、角色權限等敏感資訊
- Refresh Token 於 Redis 中記錄，登出或換發時作廢舊 Token
- Access Token **禁止**存入 localStorage（XSS 風險），前端以記憶體變數持有

### Cookie 設定（Refresh Token）

| 屬性       | 值                                |
| ---------- | --------------------------------- |
| `HttpOnly` | `True`（**強制**）                |
| `SameSite` | `Lax`                             |
| `Secure`   | 生產環境 `True`，開發環境 `False` |
| `Path`     | `/api/v1/auth`                    |

## API 端點

所有認證相關 API 位於 `backend/app/api/v1/auth/`：

| 方法   | 路徑                           | 說明         |
| ------ | ------------------------------ | ------------ |
| `POST` | `/api/v1/auth/register`        | 註冊         |
| `POST` | `/api/v1/auth/login`           | 登入         |
| `POST` | `/api/v1/auth/logout`          | 登出         |
| `POST` | `/api/v1/auth/refresh`         | 換發 Token   |
| `POST` | `/api/v1/auth/reset-password`  | 驗證身分後重設密碼 |

## 註冊

### 必填欄位

| 欄位       | 驗證規則                                                 |
| ---------- | -------------------------------------------------------- |
| 使用者名稱 | 必填，2–50 字元                                          |
| 帳號       | 必填，>= 8 字元，至少 1 個英文字母 + 1 個數字，不可重複 |
| 密碼       | 必填，>= 8 字元（詳見密碼規則）                          |
| 確認密碼   | 必填，須與密碼欄位完全一致                               |

### 密碼規則

- 長度 >= 8 個字元
- 至少包含一個大寫字母、一個小寫字母、一個數字
- 後端使用 **bcrypt** 雜湊儲存，**禁止**明文存放
- 前端於送出前進行格式驗證，後端亦須獨立驗證（不信任前端）

### 註冊流程

1. 前端驗證欄位格式（即時回饋）
2. 送出至後端 → 後端驗證帳號唯一性 + 密碼規則
3. 密碼 bcrypt 雜湊後寫入資料庫
4. 回傳成功訊息，導向登入頁面

## 登入

### 登入欄位

| 欄位   | 說明       |
| ------ | ---------- |
| 帳號   | 使用者帳號 |
| 密碼   | 使用者密碼 |

### 登入流程

1. 前端送出帳號 + 密碼
2. 後端驗證帳號是否存在、密碼是否正確（bcrypt compare）
3. 驗證通過 → 簽發 Access Token + Refresh Token
4. 驗證失敗 → 回傳通用錯誤訊息（「帳號或密碼錯誤」），**禁止**區分「帳號不存在」或「密碼錯誤」

### 登入頁面

- 掛載於 `/`（首頁），**不設獨立 `/login` 路徑**（對應 `10-frontend.md § 路由群組`）
- 登入成功後導向 `/dashboard`

---

## 重設密碼

### 重設密碼流程

1. 使用者輸入帳號＋使用者名稱
2. 後端驗證帳號與使用者名稱是否匹配
3. 驗證通過 → 使用者輸入新密碼 + 確認密碼
4. 後端驗證密碼規則 → 更新密碼（bcrypt 雜湊）
5. 驗證失敗 → 回傳通用錯誤訊息（「帳號或使用者名稱錯誤」），**禁止**區分何者不正確

### 重設密碼規則

- 新密碼須符合與註冊相同的密碼規則

## 安全性注意事項

- **禁止**在 log 中記錄密碼明文或 JWT 完整內容
- **禁止**在 API 回應的 `detail` 中揭露使用者是否存在（防止帳號列舉攻擊）
- 重設密碼時，帳號或使用者名稱驗證失敗皆回傳相同錯誤訊息（防止資訊洩漏）

### 登入失敗鎖定機制

透過 Redis 記錄同一帳號的連續失敗次數，依門檻逐級升級鎖定：

| 連續失敗次數 | 處置                                                  |
| ------------ | ----------------------------------------------------- |
| 5 次         | 暫時鎖定 15 分鐘（Redis TTL），到期後自動解鎖         |
| 10 次        | 暫時鎖定 1 小時（Redis TTL），到期後自動解鎖          |
| 15 次        | 永久鎖定：`is_active` 設 `False`，須管理員手動解鎖    |

- 鎖定期間所有登入嘗試直接拒絕，回傳通用訊息（「帳號已被鎖定，請稍後再試」），**禁止**揭露剩餘鎖定時間或失敗次數
- 登入成功後重置失敗計數器
- `is_active = False` 的帳號即使密碼正確也**禁止**登入，須由管理員透過後台恢復
