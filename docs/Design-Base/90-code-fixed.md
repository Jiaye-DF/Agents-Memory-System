# 程式碼修正記錄規範（fixed.md）

本文件規範**每次修改既有程式碼時**，需同步在對應版本的 `fixed.md` 留下修正紀錄的格式與時機。

---

## 1. 為何需要 fixed.md

- `tasks-v*.md` 是**實作前規格**，記錄「應該怎麼做」。
- `commit message` 是**事件流水**，逐筆短描。
- 兩者都不適合承載「為什麼這個 bug 會發生 / 根因 / 影響範圍 / 同類型如何避免」這類**事後可被搜尋查閱**的內容。
- `fixed.md` 補上這塊：**驗收 / 線上 / 後續開發過程中發現的 bug 與修正紀錄**，集中於版本資料夾，便於後人快速複習踩過的坑。

---

## 2. 何時必須寫入 fixed.md

只要**修改既有程式碼**並符合下列任一條件，就必須在當前進行版本的 `docs/Tasks/v{X.Y}/fixed.md` 新增一條：

- 修補 bug（功能不符預期、報錯、UI 異常）
- 修正設定 / 環境變數 / docker-compose / migration 等基礎設施錯誤
- 修正既有規格實作偏差（不含「依新版規格擴充」— 那屬於 tasks 範疇）
- 第三方相依升級 / API 變動造成的 breaking change 修補
- 效能 / 安全性回填修正

**不需寫入**的情境：

- 純粹依 `tasks-v*.md` 規格的新功能實作（屬於版本任務，回填於 tasks 文件）
- 純文件修訂、註解調整、格式排版
- 重命名 / 重構但不改變行為（除非為修 bug 而做）

---

## 3. 條目格式

每條修正以二級標題 `## N. <簡述>` 開頭，**標題旁標註修正時間**（時區：`Asia/Taipei`）：

```markdown
## 3. Anthropic structured output 不支援 maxItems / maxLength  〔2026-04-22 14:35:10〕
```

時間格式 **`YYYY-MM-DD HH:MM:SS`**，以全形〔〕包覆放在標題尾端，**不**放於各區塊或表格內。

### 條目內必備區塊

```markdown
## N. <簡述>  〔YYYY-MM-DD HH:MM:SS〕

**問題**：<現象描述。能附 log / 錯誤訊息更佳，用 ```text 區塊>

<根因說明：為什麼會發生、何時被埋下、為何先前未暴露>

**修正**：<採取的處置；多步驟可用 1./2./3. 或 bullet>

**影響檔案**：

- `path/to/file1`
- `path/to/file2`
```

依需求可加：

- `**新增欄位**` / `**API 變更**` 表格（schema 類修正）
- `**驗證方式**`（如何重現與確認已修）
- `**殘留 / 後續**`（短期 workaround、待追項）

### 範本參考

- 簡潔型：[v1.0/fixed.md §1](../Tasks/v1.0/fixed.md)（單一檔案改名衝突）
- 含根因 + log：[v1.1/fixed.md §1](../Tasks/v1.1/fixed.md)（FK 跨 metadata）
- 含三層防禦敘述：[v1.1/fixed.md §3](../Tasks/v1.1/fixed.md)（Anthropic schema）
- 純基礎設施：[v1.1/fixed.md §2](../Tasks/v1.1/fixed.md)（compose env passthrough）
- 無程式改動但仍須記錄：[v1.1/fixed.md §5](../Tasks/v1.1/fixed.md)（Turbopack cache，影響檔案寫「無」）

## 4. 檔案結構

每個版本資料夾各自維護一份 `fixed.md`：

```
docs/Tasks/
├── v1.0/fixed.md
├── v1.1/fixed.md
├── v1.2/fixed.md   ← 進入 v1.2 開發後依需要建立
└── v1.X/fixed.md
```

開頭採以下骨架：

```markdown
# v{X.Y} 修正記錄

> 一句話描述本版 fixed.md 涵蓋情境（例：驗收 v1.X.N 時陸續發現的 bug 與修正）。

---

## 1. <第一條修正>  〔YYYY-MM-DD HH:MM:SS〕
...

---

## 處理狀態

| # | 項目 | 狀態 | Commit |
| --- | --- | --- | --- |
| 1 | ... | ✅ 已修 | <hash 或 — 表示待 commit-all> |

---

## 殘留清理項

- <條列尚未處理 / 需手動執行的後續事項，例：清 DLQ、刪暫存>
```

> `## 處理狀態` 表格與 `## 殘留清理項` 為**選用**，當條目超過 3 條或有跨條目的待追事項時加入。

---

## 5. 撰寫紀律

- **時間以實際發現問題的時間**為準，不是 commit 時間；commit 可能晚於發現幾天。
- 每條只描述**一個獨立問題**；多個無關 bug 開多條，不要併段。
- **根因**段是核心 — 沒寫清楚根因的條目視為品質不合格，未來查閱無價值。
- 影響檔案用**相對於 repo 根目錄**的路徑，不用絕對路徑。
- 程式碼 / log 一律用 fenced code block 並標語言（`text` / `bash` / `sql` / `python` / `ts` ...）。
- 中文敘述、半形數字 / 英文符號之間用空格（沿用本專案 markdown 慣例）。

---

## 6. 與 tasks 文件的分工

| 場景 | 寫進 `tasks-v*.md` | 寫進 `fixed.md` |
| --- | --- | --- |
| 規格內任務完成 | ✅ 回填 `[x]` | ✗ |
| 規格內任務被改寫 | ✅ 保留 `[ ]` + `—（已改為 ...）` | ✗ |
| 完成後驗收發現 bug | ✗ | ✅ 新增條目 |
| 上線後使用者回報 bug | ✗ | ✅ 新增條目 |
| 修正涉及規格層調整 | ✅ 同步修正 tasks 描述 | ✅ 條目交叉引用 tasks |

**雙向引用**：fixed.md 條目若涉及某個 task 規格被推翻，於該條目尾段附 `見 tasks-vX.Y.Z.md §N`；對應的 task checkbox 亦於後方補 `—（已改為 xxx，見 fixed.md §N）`。

---

## 7. 與 CLAUDE.md「任務文件回填」原則的關係

CLAUDE.md 已要求「commit message 是事件紀錄，task doc 是階段真相來源」。本規範補充第三層：**fixed.md 是品質演進真相來源** — 記錄系統從「規格上應該如何運作」演進到「實際運作上踩過哪些坑、為什麼」。三者同時維護，缺一不可。

---

## 8. 跨版本既存問題的標註慣例

部分 bug / 規範違反**並非當版本引入**，而是更早版本（甚至 v1.0 起）就存在、本版本才被掃描或重構發現。這類「跨版本既存問題」仍可寫入**當版本** `fixed.md`（因為修正動作在當版發生），但條目內**必須**：

1. 條目簡述後加註 `（既存自 v{X.Y}）`，範例標題格式：`## 5. Skill 上傳缺 zip bomb 防線（既存自 v1.2.3）  〔2026-04-27 11:13:04〕`
2. **根因**段交代：
   - 該違規在哪個版本 / 哪個 PR 被引入（commit hash 或 task §N）
   - 為何先前未暴露 / 未被掃描出
3. **修正**段若橫跨多支檔案 / 多版本累積，可分點列實際改動

> 為何不寫到引入版本的 `fixed.md`：fixed.md 是**該版本的品質演進真相**，回填到舊版本會破壞「該版本完成時的當下視角」。而當版本掃描出舊規範違反並修補，仍是**當版本的處理動作**，理應記在當版。
>
> 若同類問題反覆出現（例：跨資源命名不一致、共用 helper 漏抄），於 `fixed.md` 條目最末以 `> 同類追蹤：見 vX.Y/fixed.md §N` 連結到前次條目，便於後續做模式分析。
