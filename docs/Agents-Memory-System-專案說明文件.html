Part I

系統與技術設計

這個平台解決什麼問題、長什麼樣子、記憶怎麼流、做了哪些工程決策與技術價值。

1專案摘要

把「每次對話都要重講背景、Prompt 技巧散落個人筆記」變成「Agent 自帶記憶、Skills 全公司沉澱共享」的 AI 助手平台。

對話重點由背景 Worker 自動抽取為向量化記憶並逐層聚合（Session → 專案 → 使用者），之後的對話以 Agentic RAG 自動檢索注入上下文；Agent 可由使用者自訂並組合 Skills，資源以公開 / 私有市集形式共享，全程成本入帳、操作可稽核。

憶三層記憶

chat / project / user 三層向量記憶（pgvector），自動抽取、主題聚合、長期偏好累積，檢索時 RRF 融合三層結果。

代自訂 Agent

System prompt、模型、語言、Temperature 皆可自訂，組合多個 Skills；同一對話可掛多個 Agent 協作。

集資源市集

Skills / Scripts / Agents 公開分享、收藏、下載、排行；v1.6 起支援 AI 語意查詢，用自然語言找到對的 Skill。

🔍 v1.6 新增能力 · Skills RAG 語意檢索（AI 查詢）
Skill 以 name / description / content 三向量（skill_embedding 1:N）建索引，檢索取 per-skill MAX 相似度並以實測校準門檻 0.45 過濾，再由 LLM 生成推薦理由；每次查詢（含 0 命中）寫入 skill_search_log 稽核快照。

2目的

🧠 記憶持久化

對話重點自動抽取為記憶並跨 Session / 專案累積，LLM 不再是金魚腦，使用者不必每次重講背景。

🧩 知識沉澱與複用

Prompt 技巧元件化為 Skill / Agent / Script，公開市集全公司共享；Skill 工廠更能自動從對話中學出 Skill 候選供人工審核入庫。

💰 成本可觀測

所有 LLM 呼叫經 單一計量進入點寫入 llm_call_log（token / 成本 / 延遲），並以 Classifier 三路分流讓便宜問題不走貴模型。

🔐 企業級治理

DF-SSO 單一登入 + RBAC 角色權限；下載、AI 查詢皆留稽核紀錄，資源軟刪除保留歷史、S3 檔案絕不真刪。

4核心概念與角色

理解平台前需先掌握的對話組織模型、三種資源、三層記憶與權限角色。

對話組織模型

對話以階層方式組織，記憶沿此階層逐層抽取與聚合：

專案 Project
分組 Session，聚合專案層記憶
→
Session
單一對話串，可掛多個 Agent（含 primary）
→
訊息 / 附件
SSE 串流回覆；圖片走 vision、文字檔拼入 prompt
→
記憶
背景 Worker 自動抽取 + 向量化

三種資源

Agent

可對話的 AI 角色

System prompt + 模型 + 語言偏好 + Temperature / max_tokens，並可綁定多個 Skills；支援範本快速建立與草稿自動儲存。

Skill

技能文件包（zip）

上傳 zip（可含多份 md，自動拼接進 prompt），支援線上編輯、下載、AI 語意查詢；檔案存 AWS S3。

Script

腳本 / 資料夾

可上傳整個資料夾的腳本資源，與 Skill / Agent 同樣支援公開分享、收藏、下載與標籤分類。

可見性與社群互動
三種資源皆有 private / public 可見性：公開資源全站登入者可讀、收藏、下載（計入排行）；修改限擁有者與 admin，軟刪除與切換可見性僅擁有者本人（admin 也不能代刪）。標籤（tag）為 per-user 標籤池，三種資源共用並支援列表 AND 篩選。

三層記憶

層級資料表內容與聚合方式
Session 層chat_memory單一對話的重點片段，由 memory_worker 從訊息即時抽取 + embedding（1536 維，HNSW 索引）。
專案層project_memory聚合專案下所有 session 記憶，依主題分群二次聚合，跨 session 找相關討論。
使用者層user_memory跨專案 30 天窗的長期偏好；同主題 ≥5 筆且占比 ≥60% 才聚合，避免噪音入庫。

對話時三層各撈 top-N 後以 RRF（Reciprocal Rank Fusion）融合排序，再組入 prompt——完整藍圖見 docs/Arch/00-memory-system.md。

角色與權限

資源 / 動作管理員 admin一般使用者 member
自建 Agent / Skill / Script CRUD✅✅
公開資源 讀取 / 收藏 / 下載✅✅
修改他人資源✅（擁有者 + admin）❌
軟刪除 / 切換可見性❌（僅擁有者本人）✅ 僅自己的
使用者 / 角色管理✅❌
LLM 模型清單、語言、範本、系統設定✅ 管理僅讀啟用中清單
LLM 成本 metrics / debug trace✅❌

角色由 user_role 表管理（內建 admin / member，註冊預設 member），寫入 JWT 的 role claim 即時比對；前端另有路由守衛，但實際過濾一律由後端完成。

5系統流程圖

流程 A：對話 Agentic RAG 管線（核心資料流）

一則訊息從送出到串流回覆，記憶如何被檢索與注入。

🔑 設計關鍵
整條管線只有主回答是貴的 LLM 推理，classifier / embedding / vector search 成本低 1~3 個量級；記憶抽取完全非同步，不增加對話延遲。不引入 LangChain 等框架，以 httpx + OpenRouter 自建，每個元件可控可觀測。

流程 B：記憶抽取與聚合管線（4 個背景 Worker）

Redis BRPOP 佇列驅動的背景管線，逐層把對話變成可檢索的長期記憶，並自動學出 Skill 候選。

流程 C：Skills AI 語意查詢（v1.6 系列核心創新）

把 RAG 能力從對話記憶帶到資源檢索本身——用自然語言找 Skill。

🔑 為什麼是三向量
v1.6.0 原以單一長向量（name + description + content 串接）建索引，實測發現語意被稀釋——「Coolify」單詞查詢對同名 Skill 相似度僅 0.49。v1.6.2 重設計為 skill_embedding 1:N 三向量、檢索取 per-skill MAX，並以實測校準門檻 0.45（正例 0.58~0.89 / 雜訊 ≤0.33）；向量於寫入側自動維護，歷史資料以 backfill script 補齊。

從資源管理工具的 MVP，成長為「可對話、有記憶」的 Agent 平台，再進化到三層記憶、成本可觀測與 Skills RAG 語意檢索。

v1.2 · 社群互動 + Scripts + 排行（.1–.5）

v1.3 多 Agent + 三層記憶 + 可觀測（.0–.7）

8技術價值

這份專案展現的可遷移工程能力 —— 適合作為技術能力佐證。

🧠 RAG 工程與實測校準

三層記憶 + RRF 融合、HNSW 索引；發現長向量語意稀釋後以實測數據重設計為三向量架構並校準門檻，而非憑感覺調參。嚴守 RAG 適用邊界——時序資料走 SQL、不為 vector 而 vector。

🏗️ 免框架 Agentic 架構

刻意不引入 LangChain，以 httpx + OpenRouter 自建 classifier → embedding → 檢索 → 融合 → 生成管線，每個元件成本、行為、失敗模式皆可控可觀測。

⚙️ 非同步管線設計

Redis BRPOP 佇列 + DLQ + 重試驅動 4 個背景 Worker，記憶抽取與聚合完全不擋對話回覆；佇列深度納入 healthcheck，阻塞讀取連線獨立配置避免誤報逾時。

💰 成本工程

Classifier 三路分流讓便宜問題不走貴模型，並記 counterfactual baseline 證明「省了多少」；先量再優化——任何優化前必先有 metrics。

🛡️ 認證與安全

雙 Token（Access 15 分鐘僅存記憶體 / Refresh 7 天 HttpOnly Cookie）+ Redis 黑名單；登入漸進鎖定（5 次 15 分 / 10 次 1 小時 / 15 次永久）；SSO back-channel logout 走 HMAC-SHA256 驗簽。

📦 儲存工程

本地 volume 平滑遷移 AWS S3：三領域共用 storage module、人類可讀 key、覆蓋走 bucket versioning、刪除採軟刪 tag 絕不真刪，可追溯可還原。

📈 可觀測與稽核

三本稽核帳：llm_call_log（成本 / 延遲 / 用途維度）、download_log（不可變下載紀錄）、skill_search_log（查詢快照，0 命中也記）——花多少、誰拿了什麼、搜了什麼皆可回答。

🔄 規格驅動迭代

v1.0 → v1.6 穩定演進，每版皆有 propose（設計提案）/ tasks（任務拆解）/ fixed（修正紀錄）文件鏈；Flyway V1–V61 增量 migration，規格與 schema 演進全程留痕。

💡 一句話總結
本專案完整涵蓋「記憶 × 檢索 × 成本 × 治理」四大 LLM 應用平台關注點，從 0 到 1 免框架自建 Agentic RAG 管線並以實測數據迭代，是RAG 工程、非同步架構與 AI 平台設計能力的綜合展現。

10第一次登入平台

所有人（admin 與 member）都從同一個前端登入；差別只在 Sidebar 是否出現「系統管理」分組。

  1. 開啟前端首頁 /——首頁即登入頁（不設獨立 /login）。
  2. 兩種登入方式擇一：本地帳密（先於 /register 註冊，預設角色 member）或 DF-SSO（導向公司中央認證，回調後自動換發本平台 Token）。
  3. 登入成功導向 /dashboard 儀表板；若你是 admin，Sidebar 會多出「系統管理」分組。
  4. 需要管理權限時，由現任 admin 至「使用者管理」（/admin/users）把你的角色調為 admin——角色寫入 Token，重新登入後生效。
登入機制
平台採雙 Token：Access Token（JWT，15 分鐘）僅存前端記憶體、禁存 localStorage；Refresh Token（7 天）走 HttpOnly Cookie，過期自動換發，登出即入 Redis 黑名單。連續登入失敗有漸進鎖定：5 次鎖 15 分鐘、10 次鎖 1 小時、15 次永久鎖（is_active=False，須管理員解鎖）；錯誤訊息一律通用化，防帳號列舉。

12維運核心觀念

操作前先記住這幾條，能避開九成「為什麼沒反應 / 為什麼不能刪」的疑惑。

  1. 所有 LLM 呼叫必經 call_llm_metered——這是全系統唯一計量出口，繞過即漏帳；code review 必檢 from app.clients.openrouter import 不得出現在計量層以外。
  2. 記憶是非同步的：送出訊息後由背景 Worker 抽取，不會立即出現；前端靠 SSE memory_updated 事件更新，佇列深度可由 GET /health 查看。
  3. 軟刪除為主：資源刪除保留歷史紀錄；S3 檔案絕不真刪（versioning + 刪除 tag），誤刪可還原。
  4. 軟刪 / 切換可見性僅擁有者本人——admin 可修改他人資源，但不能代刪、不能代切可見性。
  5. Migration 走 Flyway 增量鏈（V1–V61）：舊 V 檔禁止修改，任何 schema 變更只能新增 V 檔。
  6. Skill 向量寫入側自動維護：新增 / 編輯 Skill 會自動重算 embedding；歷史資料需跑 backfill script。語意檢索門檻 skill.rag.min_score 出廠 0.45，可於系統設定調整。
  7. Log 噪音用 LOG_LEVEL 控制：預設 INFO，正式部署建議 WARNING。
  8. LINE / Telegram 為架構預留：compose 有變數 passthrough，但後端尚無對應程式——目前尚未實作，勿據此排錯。

13常見需求 Playbook

照情境找對應流程，依步驟點即可。

A · 幫新成員開通帳號

情境：新同事要開始使用平台。
  1. 請對方以 DF-SSO 直接登入（公司帳號），或於 /register 註冊本地帳號——兩者預設角色皆為 member，即可建資源、對話。
  2. 若需管理權限，admin 至 /admin/users 將其角色調為 admin。

B · 建一個會用 Skill 的 Agent 並開始對話

情境：想把一套工作方法（如程式碼審查規範）變成可對話的 AI 角色。
  1. 到 /skills 上傳 Skill（zip，可含多份 md，會自動拼接進 prompt）。
  2. 到 /agents/new 建立 Agent：填 system prompt（或套範本）、選模型與語言、勾選剛上傳的 Skills。
  3. 到 /sessions/new 開新對話並選擇該 Agent；對話中可再掛入其他 Agent 協作（指定 primary）。
  4. 正常對話即可——重點會自動變成記憶，之後的對話愈來愈懂上下文。

C · 找不到合適的 Skill

情境：想找「能幫我部署專案」之類的技能，但關鍵字搜不到。
  • 在 /skills 或儀表板公開 Skills 頁籤，把搜尋框左側模式切成 AI 查詢，用自然語言描述需求送出。
  • 系統做語意檢索並附 LLM 推薦理由；查無結果代表相似度低於門檻（0.45）——換個說法再查，或自己上傳後公開分享。

D · 管理可用的 LLM 模型

情境：要開放新模型、或下架昂貴 / 停供的模型。
  • admin 至 /admin/models 新增 / 停用模型（provider 統一為 OpenRouter）；member 建 Agent 時只能從啟用中清單選。
  • 可設定 is_default 預設模型與 max_output_tokens 上限。

E · 查 LLM 花費與異常用量

情境：要知道這個月花多少、哪個人 / Agent 最燒錢。
  • 成本資料在 llm_call_log（admin metrics 端點 / SQL 查詢），可依 user / session / agent / purpose（chat、memory_extract、embedding、classifier…）維度切片。
  • Classifier 同時記 baseline 成本（假設全走貴模型會花多少），「省了多少」有數據可證。

F · 處理 Skill 工廠的建議

情境：Sidebar「Skill 建議」出現 pending 徽章。
  • 到 /skill-suggestions 逐筆審核：系統從對話記憶自動學出的 Skill 候選，採納即正式入庫成 Skill、退回即駁回。
  • 建議一律人工審核後才入庫，工廠不會自動發布。

G · 帳號異常鎖定 / 停用

情境：有人被鎖在門外，或要收回離職者的存取權。
  • 被鎖：連續錯 5 次鎖 15 分、10 次鎖 1 小時可等待自動解鎖；15 次為永久鎖（is_active=False），須 admin 至 /admin/users 解鎖。
  • 停用：admin 將帳號停用即拒絕登入；資源採軟刪除，歷史紀錄與稽核不受影響。

14使用者：對話與記憶手冊

日常使用者最常用的對話功能與「記憶怎麼運作」。

① 開始對話

  • /sessions/new 選一個 Agent 開聊；Session 可不綁專案，快速提問無負擔。
  • 要長期經營的主題，建議掛在對話專案（/projects）下——專案會聚合出自己的專案層記憶。

② 多 Agent 協作

  • 同一 Session 可掛多個 Agent（指定 primary），不同專長的 Agent 在同場對話接力。

③ 附件

  • 支援上傳附件：圖片由 vision 模型理解、文字檔內容拼入 prompt；檔案存 S3。

④ 記憶如何生效

你正常對話
不用特別「叫它記住」
→
背景自動抽取
重點寫入 Session 記憶，並逐層聚合到專案 / 個人偏好
→
下次對話自動帶上
相關記憶檢索後注入，不必重講背景

記憶抽取是非同步的，送出訊息後幾秒內完成；完成時介面會收到 memory_updated 即時更新。

⑤ 資源分享與下載

  • 自己的 Agent / Skill / Script 切成 public 即全站可見；他人可收藏、下載（計入排行與 download_log 稽核）。
  • 下載 Agent 會輸出 AGENTS.md；Skill / Script 為原始檔案包。
⚠ 帳號安全
Access Token 僅存於瀏覽器記憶體，重新整理由 Refresh Cookie 自動換發——不需要也不應手動保存 Token；共用電腦請務必登出（登出即撤銷 Refresh Token）。
