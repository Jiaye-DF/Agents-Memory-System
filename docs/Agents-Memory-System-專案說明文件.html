Part I

系統與技術設計

這個平台解決什麼問題、長什麼樣子、資料怎麼流、做了哪些工程決策與技術價值。

1專案摘要

把「每次對話都從零開始、AI 什麼都不記得」變成「Agent 會累積記憶、越用越懂你、還能從對話自學新技能」的 AI 助手平台。

使用者建立自己的 Agent（角色設定 + 掛載 Skills + 指定模型），在 Project / Session 中與其對話；系統以 三層記憶（Session / Project / User） 持久化每段對話的重點並向量化（pgvector），下次對話時以 RRF 融合檢索 把相關記憶注入 system prompt 實現 Agentic RAG。Skill / Script / Agent 可公開分享形成市集，並以 多向量語意檢索（AI 查詢）找到最適合的 Skill。

憶三層持久化記憶

背景 worker 把對話摘要成 Session → Project → User 三層記憶並向量化，跨對話、跨專案累積長期偏好。

組自訂 Agent × Skills

Agent 表單化設定（角色、語言、模型、生成參數），模組化掛載 Skills；支援多 Agent 同 Session 協作。

檢Skills RAG 語意檢索

每個 Skill 建 name / description / content 三條向量，HNSW 檢索 per-skill 取最高分，LLM 產生推薦理由。

學Agentic Skill 工廠

掃描累積的記憶，LLM 自學產出 Skill 候選供人工審核入庫，並主動推薦給適合的 Agent，形成自我成長閉環。

🧪 v1.6 系列新增能力 · Skills RAG + AI 查詢稽核
Skill 市集從關鍵字搜尋升級為語意檢索：向量儲存重設計為 skill_embedding 1:N 多向量表（v1.6.2），檢索採兩層 HNSW SQL、per-skill 取 MAX，min_score 0.45 實測校準（正例 0.58–0.89 全收、雜訊 ≤0.33 全擋）；每次 AI 查詢寫入 skill_search_log 稽核快照（0 命中也記，作為內容缺口訊號，v1.6.3）。

2目的

🧠 讓 AI 有記憶

對話重點自動摘要、向量化、持久化，跨 Session / Project 累積使用者偏好，AI 回應越用越貼合情境。

🧩 能力模組化與共享

Skills / Scripts / Agents 皆為可上傳、可版本更新、可公開分享的資源，搭配收藏 / 下載 / 排行榜形成內部市集。

🤖 系統自我成長

Agentic Skill 工廠從記憶中歸納可複用的技能，經人工審核轉為正式 Skill —— 知識從對話沉澱為資產。

💰 成本可控可稽核

所有 LLM / embedding 呼叫強制走單一計量入口寫入成本 log；前置分類器把訊息分流 skip / cheap / expensive，避免大砲打小鳥。

4核心概念與角色

理解平台前需先掌握的資源模型、記憶層級與權限角色。

資源模型

Agent

對話主體 · 可組合

角色 prompt + 語言偏好 + 指定 LLM 模型 + 生成參數，可掛載多個 Skills；支援範本快速建立、匯出 AGENTS.md。

Skill

技能元件 · ZIP 上傳

模組化技能包（ZIP 存 S3），支援整包重上傳與單檔線上編輯（樂觀鎖）；掛載後以純文本注入 Agent system prompt。

Script

腳本資源 · 檔案 / 資料夾

可上傳單檔或整資料夾的腳本資源，與 Skill 同樣支援公開分享、收藏、下載計數與標籤。

Project / Session

對話容器 · 兩層

Session 為對話單位，可歸屬 Project 或游離存在、可移入移出；一個 Session 可掛多個 Agent（1:N）並指定主 Agent。

三層記憶

Session → Project → User

背景 worker 逐層聚合：Session 記憶即時萃取，Project 記憶彙整同專案，User 記憶沉澱跨專案長期偏好。

Skill 候選

Agentic 自學產物

Skill 工廠掃描記憶產出的候選技能（含信心分數與來源記憶），須人工審核核准後才成為正式 Skill。

資源可見性

Agent / Skill / Script 皆有 私人 / 公開 兩態：私人僅擁有者可見；公開進入市集供所有人瀏覽、收藏與下載。權限判斷收斂於共用的三段式檢查（core/access.py）：可讀（owner 或公開）→ 可改（owner）→ 擁有者限定，前端僅控制顯示，實際過濾一律由後端完成。

角色與權限

資源 / 動作管理員 admin一般成員 member

使用者管理（角色 / 停用）✅❌
LLM 模型清單 / 語言 / 範本管理✅ 維護唯讀（表單下拉選取）

5系統流程圖

流程 A：對話訊息處理管線（核心資料流）

一則使用者訊息從送出到回覆，經過分類、檢索、組裝與計量的完整路徑。

🔑 設計關鍵
記憶寫入完全非同步（Redis 佇列 + 背景 worker），不阻塞對話回應；所有 LLM / embedding 呼叫強制經 llm_metering.call_llm_metered 唯一入口，skip 路線也寫 baseline log，成本紀錄不漏帳。

流程 B：三層記憶管線（背景非同步）

對話重點如何沉澱為 Session → Project → User 三層可檢索的長期記憶。

流程 C：Skills RAG 語意檢索（v1.6 系列核心創新）

「AI 查詢」如何從自然語言找到最適合的 Skill，並留下完整稽核紀錄。

🔑 設計重點
一個 Skill 拆 name / description / content 三條獨立向量（1:N 儲存於 skill_embedding），任一面向命中即可召回；min_score 0.45 為 V60 實測校準（正例 0.58–0.89 全收、雜訊 ≤0.33 全擋，邊際 > 0.12）；UI 自 v1.6.3 起隱藏相似度分數，分數轉入稽核 log 供後台分析。

8技術價值

這份專案展現的可遷移工程能力 —— 適合作為技術能力佐證。

🧠 向量檢索工程

pgvector HNSW cosine 索引、多向量 1:N 儲存設計、兩層 SQL（純 HNSW 近鄰 → join 過濾 + per-skill MAX）、以實測資料校準 min_score 而非拍腦袋定值。

🏗️ 記憶系統架構

三層記憶（Session / Project / User）逐層聚合、RRF 多路融合檢索、規則預篩 + 小模型萃取的成本分層，完整的 Agentic RAG 落地。

⚙️ 非同步管線設計

Redis 佇列 + DLQ + 4 個背景 worker，記憶寫入與對話回應完全解耦；pub/sub → SSE 讓前端即時看到記憶更新，不引入 WebSocket 的最小方案。

💰 LLM 成本工程

唯一計量入口強制所有呼叫入帳（含 skip baseline）、前置分類器分流 skip / cheap / expensive、embedding 內容截長、30 天 log 保留策略。

🤖 Agentic 自學閉環

「對話 → 記憶 → 歸納 Skill 候選 → 人工審核 → 推薦掛載」的自我成長迴路，AI 產出一律過人工閘門，兼顧自動化與可控性。

🔐 認證與安全

JWT 雙 Token（httpOnly refresh cookie）、DF-SSO OAuth2 / Azure AD bridge 與本地登入並存、back-channel logout（HMAC 驗證）、production CORS 白名單強制。

🧱 工程化實踐

Monorepo 分層架構（api / service / repository / model）、Flyway 增量 migration V1–V61（build-time 烤 image 保部署一致）、S3 tag 軟刪不物理刪除、統一 ApiResponse 契約、Swagger 文件。

🔄 規格驅動迭代

v1.0 → v1.6.3 穩定演進，每版皆有 propose（設計提案）與 tasks（任務拆解）文件並回填進度，架構重設計（v1.6.2）敢於推翻既有欄位設計並校準驗證。

💡 一句話總結
本專案完整涵蓋「記憶 × 檢索 × 成本 × 自學」四大 AI 平台關注點，從 0 到 1 設計並迭代一套三層向量記憶 + Agentic RAG 系統，是向量檢索工程、非同步架構與 LLM 應用整合能力的綜合展現。

9環境與啟動

本機以 Docker Compose 一鍵啟動；正式環境由 Coolify 以 docker-compose.yml 部署並反代發佈。

本機開發環境

  1. 複製環境變數範本：cp .env.example .env，依序填入 PostgreSQL 帳密、SECRET_KEY（≥32 字元）、OPENROUTER_API_KEY、AWS S3 憑證、Flyway 帳密等必填項。
  2. 啟動：docker compose -f docker-compose.dev.yml up -d（或用自訂指令 /dev-up 一鍵啟動）。
  3. Flyway 會先完成 migration（V1–V61），成功結束後 backend 才會啟動。
服務位置
前端http://localhost:3000
後端 APIhttp://localhost:8000/api/v1
API 文件（Swagger）http://localhost:8000/api/docs
PostgreSQLlocalhost:5433（容器內 5432）
Redislocalhost:6379

正式部署（Coolify）

  • 部署分支為 main（v1.4 起）；服務組成：postgres（pgvector/pg17）、redis、flyway（一次性 migrate）、backend、frontend、adminer，全走 expose 由 Coolify 反代。
  • Migration SQL 於 build 時烤進 flyway image（runtime bind mount 在 Coolify 不可靠），backend depends_on flyway 成功完成。
  • S3 bucket 依環境分離：測試 agents-platform-test-datas / 正式 agents-platform-prod-datas。
  • 部署環境建議設 LOG_LEVEL=WARNING 抑制 INFO 噪音（預設 INFO）。
⚠ 環境變數要點
SECRET_KEY 少於 32 字元後端啟動即報錯；production 下 CORS_ORIGINS 禁止 * 與 localhost；NEXT_PUBLIC_API_URL 於前端 build 階段內聯，改值需重 build。LINE_CHANNEL_ACCESS_TOKEN / TELEGRAM_BOT_TOKEN 為預留變數，目前無對應功能。

10第一次登入

支援本地帳密與 DF-SSO 兩種登入方式並存；角色分 admin 與 member。

  1. 開啟前端網址，未登入會停在登入頁；可註冊本地帳號（Email + 密碼），或點 DF-SSO 登入走公司單一登入。
  2. 登入後採 JWT 雙 Token：access token 15 分鐘、refresh token 7 天（httpOnly cookie，僅送 /api/v1/auth 路徑）；前端自動刷新，一般使用無感。
  3. 新帳號預設角色 member，可管理自身 Agents / Skills / Scripts 與對話；需要管理功能（使用者、模型清單、系統設定）由現任 admin 在「使用者管理」調整角色。
  4. SSO 登出支援 Single Logout：在 SSO 端登出會透過 back-channel 通知本系統同步失效。

12維運核心觀念

操作前先記住這幾條，能避開九成「為什麼沒反應 / 找不到」的疑惑。

  1. 記憶是非同步產生的：對話結束後由背景 worker 摘要 + 向量化，數秒到數十秒後才出現在記憶側欄（SSE 會自動推播更新），不是即時同步。
  2. 不是每句話都會變記憶：規則預篩會略過寒暄、emoji 等低價值訊息；分類器也會讓此類訊息 skip 不呼叫 LLM。
  3. AI 查詢有品質門檻：相似度低於 0.45 一律不回傳 —— 查無結果代表真的沒有語意相近的 Skill，而非系統故障；該查詢仍會寫入稽核 log。
  4. Skill 向量自動維護：建立 / 更新 / 重上傳 Skill 時自動重建三條向量，無須手動操作；歷史資料才需要跑 backfill script。
  5. 多為軟刪除：資源刪除保留歷史（S3 檔案以 tag 標記 status=(deleted)，絕不物理刪除），計數與稽核可追溯。
  6. 所有 LLM 花費都有帳：任何繞過 llm_metering 直呼 OpenRouter 的寫法都是違規；查成本一律以 llm_call_log（30 天保留）為準。

13常見需求 Playbook

照情境找對應流程，依步驟點即可。

A · 建立第一個 Agent 並開始對話

情境：新使用者想要一個懂特定領域的 AI 助手。
  1. 「Agents」→ 新增 Agent：填名稱與角色 prompt（可先套用範本再修改），選語言偏好與 LLM 模型，必要時調 temperature / max_tokens preset。
  2. 在表單的 Skills 多選掛上需要的技能（可先用 Playbook C 的 AI 查詢找 Skill）。
  3. 到「對話」建立 Session（可先建 Project 分類，或直接建游離 Session）→ 選這個 Agent → 開始對話（SSE 逐字回覆）。
  4. 對話幾輪後，右側記憶側欄會陸續出現系統萃取的記憶條目；之後的對話會自動檢索注入相關記憶。
  5. 需要多位專家協作時，可在 Session 內加掛其他 Agent（1:N）並切換主要回應者。

B · 上傳 / 更新一個 Skill

情境：想把整理好的技能包分享給自己或全平台的 Agent 使用。
  1. 「Skills」→ 上傳：填名稱 / 描述、選公開或私人、上傳 ZIP（上限 50MB）。內容建議含 SKILL.md（語意檢索會優先讀取它建立 content 向量）。
  2. 系統自動為 name / description / content 建立三條向量，立即可被 AI 查詢召回。
  3. 小改：進 Skill 詳情用單檔線上編輯；大改：用整包重上傳。兩者觸發前都會顯示「被 N 個 Agent 使用中」提醒影響面。
  4. 若編輯時回 409，代表有人先改過（樂觀鎖），重新載入最新版再改。

C · 用 AI 查詢找 Skill

情境：不知道確切名稱，只能用一句話描述需求。
  1. 「Skills」列表或儀錶板公開市集，把 SearchModeBar 切到 AI 查詢。
  2. 用自然語言描述需求（例：「幫我把會議逐字稿整理成紀要」）→ 按查詢。
  3. 結果依語意相似度排序，附 AI 徽章與推薦理由（介面不顯示分數）；點卡片進詳情即可掛載到 Agent。
  4. 查無結果代表平台目前沒有語意相近的 Skill —— 這筆查詢已寫入稽核 log，admin 可據此補充內容。

D · 審核 Agentic Skill 候選

情境：Skill 工廠從對話記憶中自學出了新技能候選。
  1. 進「Skill 建議」頁，檢視候選的內容、信心分數與來源記憶。
  2. 核准：候選轉為正式 Skill 入庫（自動建向量、可被檢索與掛載）；駁回：丟棄。候選以 Redis 暫存（TTL 7 天），逾期未審自動失效。
  3. 對話中若收到 Skill 推薦提示（SSE 事件），也可直接就地核准 / 駁回。

E · admin：新增可用模型 / 調整系統參數

情境：要開放新的 LLM 模型給大家選用，或調整 RAG / 記憶行為。
  • 新增模型：「系統管理 → 模型」可瀏覽 OpenRouter catalog 挑選加入，設定顯示名稱、預設模型與 max_output_tokens；member 端表單下拉立即可選。
  • 調整參數：「系統管理 → 設定」維護 key-value 系統設定，涵蓋三層 RAG 門檻（top_k / min_score）、記憶萃取模型、分類器規則門檻、Skill RAG（skill.rag.*）等；改門檻前先看 llm_call_log 的 rag 命中數據再動手。
  • 使用者管理：調整角色（admin / member）、停用帳號。

F · 查成本與稽核紀錄

情境：想知道 LLM 花了多少錢、誰下載了什麼、大家都在搜什麼。
  • LLM 成本：admin GET /api/v1/admin/metrics/cost 彙總成本 / 延遲 / 失敗；明細在 llm_call_log（含 RAG 命中數與最高分，保留 30 天）。
  • 記憶 pipeline 追蹤：admin debug 端點可 trace 單一 Session 的記憶產生歷程與檢索結果，佇列 / DLQ 深度見 GET /health。
  • 下載稽核：download_log 表記錄誰在何時下載了什麼（含名稱快照），目前以 SQL 查詢、無 UI。
  • AI 查詢稽核：skill_search_log 表含查詢語句與 JSONB 結果快照（相似度分數在此），0 命中也記錄，是內容缺口的第一手訊號。

agent / agent_skillAgent 設定與 Skill 掛載中介表；含收藏 / 下載計數
